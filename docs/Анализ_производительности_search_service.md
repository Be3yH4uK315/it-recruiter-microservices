# Анализ производительности search-service

## Контекст

По результатам актуального контрольного прогона качества поиска в инфраструктуре Docker были получены следующие средние времена ответа:

- `lexical`: `107.598 ms`
- `vector`: `661.38 ms`
- `hybrid`: `1027.974 ms`

При этом качество ранжирования осталось лучшим именно у `hybrid`-режима:

- `nDCG@5` для `lexical`: `0.841313`
- `nDCG@5` для `vector`: `0.889475`
- `nDCG@5` для `hybrid`: `0.983778`

## Результаты A/B-сравнения rerank_top_k

Для выбора размера окна переранжирования был выполнен отдельный эксперимент на контрольной выборке `services/search/scripts/quality_cases.vkr.json` с вариантами `rerank_top_k = 10, 15, 20, 30`.

Базовые режимы на этой выборке показали следующие результаты:

- `lexical`: `195.151 ms`, `nDCG@5 = 0.841313`
- `vector`: `315.052 ms`, `nDCG@5 = 0.889475`

Для гибридного режима все проверенные значения `rerank_top_k` дали одинаковые метрики качества:

- `precision@5 = 0.56`
- `recall@5 = 0.933333`
- `MRR = 1.0`
- `nDCG@5 = 0.983778`

Практический вывод следующий:

- так как качество при `rerank_top_k = 10` не уступает `15`, `20` и `30`;
- а стоимость `cross-encoder` растет с увеличением числа кандидатов на входе;
- в рабочей конфигурации целесообразно использовать `rerank_top_k = 10`.
